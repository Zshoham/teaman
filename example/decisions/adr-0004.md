## Context

The first external integrators want a predictable, cacheable HTTP surface. GraphQL is attractive but the team has no operational experience running it, and our access patterns are still in flux.

## Decision

Expose a versioned REST API with conventional resources and HTTP caching. Defer GraphQL until client query patterns are well understood.

## Consequences

- Familiar tooling and edge caching work out of the box.
- Clients over-fetch on nested views and make many round trips.
- This decision was later revisited as the web client grew — see ADR-0011.
