date: 2023-02-02

tags: [repo, tooling]
summary: "Keep all services and shared libraries in one repository with a unified build graph."
---

## Context

We have three services and a shared design library. Coordinating changes across separate repos already means brittle version pinning and lockstep releases for what is effectively one product.

## Decision

All first-party code lives in a single repository. A task runner with content-addressed caching builds and tests only the affected projects per change.

## Consequences

- Atomic, cross-cutting changes land in one pull request.
- CI must learn to scope work to the affected graph or build times will balloon.
- Third-party forks and unrelated experiments stay out of the tree.
